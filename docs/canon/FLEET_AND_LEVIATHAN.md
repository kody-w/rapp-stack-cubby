# Fleet and Leviathan profile

## Two distinct meanings

**Fleet Leviathan** coordinates multiple enrolled brainstem bodies. It would
sign one bounded operation per destination, verify independent responses, and
aggregate partial failure without replaying completed effects.

**Wrapped-organism Leviathan** packages one multicellular being for movement
between environments. Its package and trust boundary are not fleet authority.
The shared name does not imply compatible protocols or artifacts.

A future fleet may map only allowlisted operations onto each node's local
`/chat`. Enrollment, destination identity, process ownership, signature,
nonce, and response binding are mandatory. Shell, deploy, arbitrary import,
MCP execution, and direct-agent routes are outside this profile.

## Status distinctions

| Class | State |
|---|---|
| **Profile requirement** | Keep fleet and wrapped-organism concepts separate and prevent either from expanding v1 authority. |
| **Implemented now** | No fleet or wrapped-organism runtime; local controller manages isolated children on one host. |
| **Mapped/reference only** | Fan-out, intermittent frames, hubs, factories, organism eggs, and multi-host supervision. |
| **Unsafe/deprecated** | Unauthenticated direct-agent routes, fleet-wide shell/deploy, weak node identity, unsigned fan-out, and manifest-controlled hatch paths. |
| **Future owner** | None in v1; requires signed twin-chat completion and a separate threat model/ADR. |
