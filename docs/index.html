Security and privacy

Public understanding, zero private state

The Pages output contains public source facts only. It cannot chat, start a twin, enroll an owner, collect input, retain browser state, or reach a private runtime. Understanding remains available with scripting disabled because the core experience is semantic HTML and CSS.

  • No backend or privileged control surface
  • No forms, analytics, third-party assets, or executable resources
  • No owner identifier, message, key, pairing, receipt, or runtime path
  • No service worker, browser persistence, or source map
  • Release hashes delegated to the external SHA256SUMS sidecar

Fresh install and hatch

Start from a fresh fork without skipping proof

Use CPython 3.11 on the declared macOS arm64 profile. Every mutable path is explicit, private, and outside the checkout. Fetch is separate; bootstrap and demo are offline.

  1. 1

    Read the handoff

    Begin with AI_CONTEXT.md, then the system model, implementation status, gap register, and operations handoff.

  2. 2

    Fetch exact inert inputs

    PYTHON=/absolute/python3.11 \
RAPP_DEPENDENCY_CACHE=/absolute/cache \
  scripts/fetch-dependencies.sh
  3. 3

    Bootstrap offline

    scripts/bootstrap-development.sh \
  --python /absolute/python3.11 \
  --venv /absolute/venv \
  --dependency-cache /absolute/cache \
  --work-dir /absolute/work \
  --install-dir /absolute/installs \
  --controller-dir /absolute/controller
  4. 4

    Run the whole product demo

    scripts/demo-product.sh --python /absolute/venv/bin/python \
  --work-dir /absolute/work \
  --dependency-cache /absolute/cache \
  --install-dir /absolute/installs \
  --controller-dir /absolute/controller \
  --receipt /absolute/demo-receipt.json \
  --cleanup
  5. 5

    Review the content-free receipt

    The demo proves deterministic build, trusted development release, egg and install integrity, authenticated controller adoption, signed-only offline SelfTest, stop/archive/unarchive, no orphan, and cleanup. It never publishes, enrolls, or sends a message.

Ten showcase prompts

Ask the repository to prove the product

These prompts are demonstrations of repository-local work, not a chat form or hosted service.

  1. One idea to public product.

    Trace one idea through reviewed source, CUBBY, rapplication, egg, controller, isolated twin, signed twin-chat, owner iMessage, and exact-commit release preparation.

  2. Map the whole RAPP system.

    Explain every selected local stage and clearly separate implemented components from mapped neighborhood, fleet, and cloud references.

  3. Verify this repository.

    Run the context, catalog, Pages, privacy, unit, and repository gates and summarize only evidence produced by those checks.

  4. Build the deterministic artifact chain.

    Fetch only locked inert dependencies, build twice with one revision and epoch, and prove every output byte is equal.

  5. Hatch an isolated twin.

    Verify a CUBBY egg, hatch it offline into separate workspace and state roots, inspect its receipt, and stop before starting services.

  6. Explain signed twin-chat.

    Walk through identity binding, P-256 signatures, freshness, durable replay handling, nonce binding, and signed responses without exposing private state.

  7. Prepare owner-only iMessage.

    Follow the fresh-fork tutorial through prerequisites and content-free preflight, leaving enrollment and live sending for the release operator.

  8. Inspect capability ownership.

    Use the implementation and capability matrices to show who owns all selected capabilities and why reference-only entries are not local claims.

  9. Trace evidence to implementation.

    Start with one direct-evidence capability, follow its context route, and identify its tested code, schema, decision, runbook, and remaining gap.

  10. Prepare an exact-commit release.

    Validate version and tag, build and hatch twice from one clean commit, prepare external sidecars, and stop before publication unless every gate is proven.
